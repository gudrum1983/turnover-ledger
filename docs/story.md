• Для FSD я бы держал stories максимально близко к компоненту, а не в общем src/stories. В твоей структуре это выглядит так:

- src/shared/ui/button-base/ButtonBase.vue
- src/shared/ui/button-base/ButtonBase.stories.ts

И аналогично для:

- shared/ui/_ и shared/lib/_, если это UI-хелперы
- widgets/_/ui/_
- pages/_/ui/_, если нужен визуальный сценарий страницы
- app/\* обычно не сторят, кроме очень инфраструктурных вещей

Главное правило по слоям:

- shared/ui: сторисы обязательны почти для каждого переиспользуемого UI-компонента
- widgets: сторисы полезны для собранных блоков в разных состояниях
- pages: сторисы только как “screen states”, не как замена e2e
- app: лучше не засорять, здесь не про изолированный UI

Что я бы рекомендовал именно тебе:

1. Удалить/вынести стандартные демо из src/stories/\*, чтобы они не смешивались с реальными компонентами.
2. Писать co-located stories рядом с компонентом.
3. Вынести общую инициализацию Storybook в .storybook/preview.ts: подключить main.css, Pinia, i18n, глобальные директивы (maska, autofocus), и при необходимости vue-router.
4. Для компонентов с зависимостями на store/router/i18n делать это через global decorators, а не дублировать setup в каждой story.
5. Для данных использовать local mocks/fixtures рядом со слоем, например model/**mocks** или lib/storybook.

Хороший практический шаблон для FSD:

- одна story = одно пользовательское состояние
- названия состояний, а не технических вариантов: Default, Loading, Empty, Error, WithValue, Readonly
- args только для внешнего API компонента
- если состояние собирается не только через props, делай render с оберткой, а не проталкивай лишние props в компонент

Для твоего проекта особенно важно сразу настроить Storybook как “mini app”, потому что у тебя компоненты уже завязаны на i18n, Pinia и директивы из src/app/main.ts. Иначе сторисы начнут ломаться или
появится соблазн писать их только для самых примитивных компонентов.
